# 功能规格：M4 指标底座（workflowhub metrics foundation）

**功能名**: `m4-metrics-foundation`
**来源用户故事**: agenthub-extraction-program roadmap M4
**需求权威源**: `artifacts/decision-log.md`（18 决策，用户 2026-06-23 批准）
**状态**: 草稿

## 速读卡（30秒看懂这个需求）

**一句话需求**：给 workflowhub 装一套自动记账本，把每个技能、每个阶段干活的执行效果记成可跨任务查询的结构化记录，供将来判断"该优化哪个技能/阶段"。

**核心改动点**（3-5 条）：
- 每次技能执行 / 每个阶段，自动采集一组核心效果字段（用量/耗时/返工/人工介入等），任务级 + 全局双写。
- 同一次执行的记录分三个时机逐步补全，最终汇成一条记录（非多条事件）。
- 卡点（返工/重试/人工升级/各类已知失败模式）记成结构化知识卡片，作为重要软反馈。
- 一份统一执行记录把进度/事实/指标/反馈/边界决策/回溯索引串成任务全景。
- 配套技能版本号规则、基线字段定义、CI 冒烟。

**最大影响面**：全为 workflowhub 新增能力，不改 agenthub 既有实现；唯一新增的对外只读关系是基线对照读取历史已归档任务记录。

**验收信号**：跑一个技能，任务级与全局两处各出现一条该次执行记录；缺任一处即判失败。指标/卡点/版本均不作拦截推进的硬门。

## 1. 问题陈述

workflowhub 框架已建好核心引擎、窄契约、配置（M1-M3 已合并），但每次技能/阶段干活的执行效果没有被记录下来——无法回答"哪个技能最慢、哪个阶段返工最多、哪一步老卡人工、该优化谁"。现有零散的工作流问题台账只在出问题时记一句话，既不结构化、也无法跨任务聚合、更没有耗时/用量等量化字段。缺这套记账底座，后续的优化决策（M14 消费迭代）就没有数据可依。

## 2. 背景、目标和边界

### 背景

agenthub 历史里有过若干可借鉴的采集件（原子追加写、拒写假零标缺口、复合键基线等做法），但都散在 agenthub 内、绑死宿主、未成体系。workflowhub 作为抽离出的独立框架，需要从零搭一套解耦宿主、可跨任务/跨项目聚合的指标底座。本期只做"底座"——采集、双写、统一记录、卡点软反馈、版本规则、基线字段。

### 目标

- 做完后，运维/优化决策者能按"技能名 + 版本"跨任务聚合查到某技能某版本的真实执行表现。
- 做完后，每个任务有一份能一眼看全的执行全景，且所有指标/卡点只记录、绝不拦阻推进。

### 非目标（明确不做）

- 不做趋势分析、优先级排序器、消费迭代入口（属后续里程碑 M14）。
- 不做自动契约比对（schema-diff）版本守卫。
- 不修改 agenthub 既有实现。

## 3. 用户场景和用例

> 这里的"用户"= workflowhub 框架的运维/优化决策者；"系统"= 指标底座。

### 场景一：技能执行被双写采集

Given 一个技能在某任务的某阶段执行，When 它执行完，Then 系统在该任务自己的目录和一个全局汇总处各产生一条该技能本次执行的记录；缺任一处即视为采集失败。

### 场景二：同一记录三时机补全

Given 一次技能执行，When 技能开始 / 技能结束 / 所在阶段结束三个时机先后到达，Then 系统对同一条记录分别落骨架、补自身结果、补"对整个阶段的影响"，三时机最终汇成同一条记录（凭执行标识定位），而非三条分散事件。

### 场景三：跨任务按技能版本聚合

Given 全局汇总已积累多任务记录，When 运维者按"技能名 + 版本号"查询，Then 能聚合出该技能该版本在哪些任务里跑过、表现如何，返回非空结果。

### 场景四：统一执行记录全景

Given 一个任务执行中，When 查看其统一执行记录，Then 能看到进度/事实/指标/反馈/边界决策/回溯索引六类键串成的任务全景，且能通过结构校验；缺任一类键即不合格。

### 场景五：卡点记成知识卡片

Given 执行中出现卡点（返工/重试/人工升级/各类已知失败模式），When 卡点出现哪怕一次，Then 系统记一张结构化卡点知识卡片，含类型、位置、根因、解决方式、是否真解决等，作为重要软反馈，不设触发阈值。

### 场景六：基线对照与缺口诚实标记

Given 已归档的 AgentHub M1-M3 任务记录，When 按字段映射算基线参照值，Then 能算的字段给出数值、跨体系算不出的字段结构化标记为缺口（含状态/原因/缺什么），并记录所用的源任务集合，绝不伪造数字或以零充缺。

### 场景七（边界）：无审查阶段的技能

Given 某技能执行流程不含审查环节，When 采集"返工轮次"字段，Then 系统须能区分"没有审查"与"审查通过零返工"两种情况，不混为一谈。

### 场景八（边界）：阶段重开去重

Given 同一阶段因故重开产生多个执行窗口，When 汇总按阶段计的字段，Then 不得跨窗口重复累加（须阶段内先去重再跨阶段求和），防止数值翻倍。

### 场景九（边界）：历史字段缺失

Given 历史任务记录本就没有某字段（如技能版本号），When 基线对照读到它，Then 标为缺口并说明"缺什么、补上数据源后能否推算"，不空占字段。

### 场景十（失败）：采集写入失败不拦推进

Given 采集写盘失败（磁盘满/权限/路径错），When 该技能或阶段仍在推进，Then 系统发出可观测告警但不得拦阻推进（守"只记不挡"铁律），且事后能从记录缺口发现这次失败。

### 场景十一（失败）：会话压缩中途重定位

Given 长会话被压缩、技能记录的中途引用丢失，When 需要继续补全该条记录，Then 系统凭执行标识从已落盘记录重新定位并续写，不依赖会话内存。

### 场景十二（失败）：数据源不可达标缺口

Given 某指标依赖的来源（如会话用量来源）在当前环境拿不到可达数据，When 采集该字段，Then 标为缺口而非填入伪造值或零值；依赖此类来源的字段在设计期须先确证来源可达再纳入。

### 状态覆盖清单

| 状态 | 含义 | 覆盖场景 |
|------|------|---------|
| 正常采集 | 技能执行被双写、三时机补全 | 场景一/二/三 |
| 全景串联 | 统一记录六类键齐全 | 场景四 |
| 卡点 | 出现摩擦记知识卡片 | 场景五 |
| 基线对照 | 历史算基线、缺口标记 | 场景六/九 |
| 无审查 | 返工字段无意义需区分 | 场景七 |
| 阶段重开 | 去重防翻倍 | 场景八 |
| 写入失败 | 告警但不拦推进 | 场景十 |
| 会话压缩 | 凭标识重定位续写 | 场景十一 |
| 源不可达 | 标缺口不填零 | 场景十二 |

## 4. 功能需求

> **溯源规则**：标 `来源：decision-log 决策 N（新增类）` 的需求，对应 grill 期用户纠偏引入的新增决策，已逐条注明。

#### 叙述层（给人读）

指标底座干三件事串成一条链：**采。** 每次技能执行和每个阶段，系统在它干活的当口自动把执行效果记下来——分三个时机（开工落骨架、收工补自身、阶段收尾补全局影响）逐步把同一条记录填满，记完往两处放：任务自己的目录（单任务回溯）和一个全局扁平汇总（跨任务聚合）。**串。** 每个任务有一份统一执行记录，不重复存原始明细，而是把进度、硬指标、软反馈、边界决策用回溯索引串成一份能一眼看全的全景。**记摩擦。** 干活中只要出现返工、重试、人工升级或各类已知失败模式，就记一张结构化卡点知识卡片，带根因和是否真解决——这是给将来优化用的重要软反馈。整条链有一条铁律贯穿始终：只记录、不拦截推进；记不全、写失败都只告警不卡流程。配套还有技能版本号规则（让聚合能区分版本）、基线字段定义（拿历史算及格线）、CI 冒烟（守住结构和聚合不退化）。

#### 编号字段层（给机器追溯）

### 采集与存储（COLLECT）

- **FR-COLLECT-001**：系统须以"每次技能执行 / 每个阶段"为粒度采集指标，非任务级粗粒度。来源：decision-log 决策 2（新增类，grill 纠偏）。
  - **场景**：Given 一个技能执行，When 采集，Then 产出粒度落到该次技能执行而非整任务一条。
  - **场景**：Given 一个阶段含多次技能执行，When 采集，Then 每次执行各一条记录，不被并成任务级一条。
- **FR-COLLECT-002**：每条执行记录须含一组核心字段：执行标识、技能或阶段名、所在阶段、技能版本、是否执行、用量、耗时、返工轮次、人工介入、卡点引用；执行标识为必填键，用于关联三时机更新与回溯索引。来源：decision-log 决策 2/5（新增类）。
  - **场景**：Given 一条执行记录，When 校验，Then 核心字段齐全且执行标识非空。
  - **场景**：Given 执行标识缺失，When 校验，Then 判不合格。
- **FR-COLLECT-003**：采集须分三时机对同一条记录逐步更新——技能开始落骨架、技能结束补自身结果、阶段结束补该技能对阶段的全局影响。来源：decision-log 决策 4（新增类，grill 纠偏）。
  - **场景**：Given 技能开始，When 采集，Then 落一条仅含骨架的记录。
  - **场景**：Given 阶段结束，When 回填，Then 同一条记录补上"对阶段影响"字段，不新建第二条。
- **FR-COLLECT-004**：记录形态为"一条记录三段更新"，非多条事件分散追加；更新须凭执行标识从已落盘记录重新定位目标记录后合并写回，不依赖会话内存中的记录引用。来源：decision-log 决策 5（新增类，grill 纠偏）。
  - **场景**：Given 会话压缩后内存引用丢失，When 续写记录，Then 凭执行标识从磁盘重定位该条并合并。
  - **场景**：Given 同一执行三时机，When 查询，Then 只得一条记录而非三条。
- **FR-COLLECT-005**：采集核心须与执行宿主解耦——核心只接收"哪个技能 / 什么时机 / 什么数据"即可记录，宿主特定的触发方式只作为可替换的适配层。来源：decision-log 决策 6（新增类，grill 纠偏）。
  - **场景**：Given 采集核心，When 检查依赖，Then 不含任何特定执行宿主钩子的硬绑定。
  - **场景**：Given 更换执行宿主，When 只替换适配层，Then 采集核心不改即可工作。
- **FR-COLLECT-006**：存储为双写——任务级 + 全局扁平汇总；全局每条记录须带任务标识、项目标识、技能名、版本号。
  - **场景**：Given 一次技能执行，When 采集完成，Then 任务级与全局各出现一条；删任一处判失败。
  - **场景**：Given 全局一条记录，When 校验，Then 含任务/项目/技能/版本四标识。
- **FR-COLLECT-007**：全局汇总的存放位置默认在用户级目录、不纳入版本库，且位置须可配置；项目标识默认从任务身份信息推导。
  - **场景**：Given 默认配置，When 写全局，Then 落用户级目录且不进版本库。
  - **场景**：Given 配置了自定义位置，When 写全局，Then 落到配置位置。
- **FR-COLLECT-008**：采集为纯记录行为，不调用模型、不引入推进开销。
  - **场景**：Given 采集执行，When 观察，Then 无模型调用发生。

### 统一执行记录（EXECREC）

- **FR-EXECREC-001**：每个任务须有一份独立的统一执行记录实体，含六类键：进度、事实、指标、反馈、边界决策、回溯索引；须能通过结构校验，缺类即不合格。来源：decision-log 决策 9（新增类，grill 纠偏）。
  - **场景**：Given 统一执行记录，When 结构校验，Then 六类键齐全则通过。
  - **场景**：Given 缺"反馈"键，When 校验，Then 判不合格。
- **FR-EXECREC-002**：统一执行记录不重复存储原始明细，而是凭回溯索引串联指标记录、软反馈与进度。
  - **场景**：Given 统一记录，When 检查指标内容，Then 是对指标记录的索引引用而非明细拷贝。
- **FR-EXECREC-003**：硬指标与软反馈分属不同数据形态分别采集，统一执行记录只做串联视图，不把两类信息混入同一条明细。来源：decision-log 决策 8（新增类）。
  - **场景**：Given 硬指标与卡点卡片，When 落盘，Then 分属不同形态，统一记录只索引不混存。
- **FR-EXECREC-004**：边界决策与回溯索引两类键须在设计期先确证数据来源（候选：现有过程记录序号作回溯索引主键、阶段/状态转移作边界决策）；确证无来源的键须显式标缺口，不得空占、不得改为人工填写。来源：decision-log 决策 10（design 前置约束）。
  - **场景**：Given 某类键无确证数据源，When 落盘，Then 显式标缺口而非空占或人工填。
- **FR-EXECREC-005**：统一执行记录每类键须标注"第一个真实消费方是谁"，无消费方的键降为可选、不强制非空，并预留扩展位以防后续里程碑发生破坏性变更。来源：decision-log 决策 18（新增类）。
  - **场景**：Given 某类键无消费方，When 校验，Then 该键为可选，不因空而 BLOCK。

### 软反馈与卡点（FEEDBACK）

- **FR-FEEDBACK-001**：卡点须以结构化"卡点知识卡片"记录，卡点出现一次即记，不设触发阈值门槛。来源：decision-log 决策 11（新增类，grill 纠偏）。
  - **场景**：Given 一次返工，When 记录，Then 即生成一张卡点卡片，不因"只一次"被略过。
- **FR-FEEDBACK-002**：卡点卡片须含必填项：卡点类型、发生位置、根本原因、解决方式、是否真正解决、发生时间。来源：decision-log 决策 11（新增类）。
  - **场景**：Given 一张卡点卡片，When 校验，Then 六个必填项齐全则通过；缺任一项判失败。
- **FR-FEEDBACK-003**：卡点类型须为受控枚举（一组从历史教训归纳的已知类型 + 其他兜底类），可选项含可迁移避坑建议、关联历史教训、"绕过非根治"标记、关联指标引用、受影响交付物。来源：decision-log 决策 11（新增类）。
  - **场景**：Given 卡点类型取枚举外的值，When 校验，Then 判不合格。
  - **场景**：Given 一个未知类卡点，When 记录，Then 可落"其他"兜底类。
- **FR-FEEDBACK-004**：卡点卡片为独立结构，作为统一执行记录"反馈"类键的内容来源之一；它与既有工作流问题台账的合并或共存方式属实现细节。
  - **场景**：Given 卡点卡片，When 统一记录引用反馈，Then 卡片是反馈键的来源之一。

### 版本与基线（VERSION / BASELINE）

- **FR-VERSION-001**：须有一份技能版本号管理规则文档，先声明"技能的对外契约 = 其清单（名称/参数/描述行为/输出）"，再界定何种变更触发何级版本变化（结构变更、行为变更、新增可选项、文档符合性修正分别对应不同级别），并规定 1.0 之前的版本处理与每次变更配套变更说明。来源：decision-log 决策 13。
  - **场景**：Given 一次技能契约变更，When 依规则判定，Then 能得出该升哪一级版本，且他人复算得同一结论。
- **FR-VERSION-002**：技能版本号须作为字段进入执行记录；本期不做自动契约比对守卫。来源：decision-log 决策 13。
  - **场景**：Given 执行记录，When 检查，Then 含技能版本字段；系统不含自动 schema-diff 守卫。
- **FR-BASELINE-001**：须有一组基线字段定义；原任务级流程指标（漏步骤/测试执行/审查执行/返工/缺陷）作为派生项，从执行记录核心字段聚合算出，不单独采集。来源：decision-log 决策 3/12。
  - **场景**：Given 核心字段就位，When 聚合，Then 原 5 项任务级指标可派生算出，无需单独采集口径。
- **FR-BASELINE-002**：基线的数据源须明确为**已归档的 AgentHub M1-M3 任务记录**（非任意"历史任务记录"）；跨体系（AgentHub→workflowhub）算不出的字段须结构化标记为缺口（含缺口状态、原因、缺什么、补上数据源后可否推算）；基线输出须记录所采用的源任务集合，并标清"准实验参照"的局限。来源：decision-log 决策 12。
  - **场景**：Given 历史归档记录，When 算基线，Then 能算的给数值、算不出的给结构化缺口。
  - **场景**：Given 跨体系不可比字段，When 输出基线，Then 标局限，不当精确对照宣称。

### 守恒与防错（GUARD）

- **FR-GUARD-001**：所有指标、卡点、版本字段均不得作为拦截推进的硬门；写入失败可发可观测告警但不拦推进。来源：decision-log 决策 15。
  - **场景**：Given 采集写入失败，When 推进，Then 发告警但技能/阶段照常推进。
  - **场景**：Given 扫描全部指标/卡点/版本相关门，When 统计，Then 作为拦截推进的硬门数量为零。
- **FR-GUARD-002**：动作计数类指标须按动作自身标识去重，不得按消息标识去重。来源：decision-log 决策 16（历史坑）。
  - **场景**：Given 同一回合多动作共享一个消息标识，When 计数，Then 按动作自身标识去重，计数不归零。
- **FR-GUARD-003**：按阶段汇总的字段在汇总前须先在阶段内去重再跨阶段求和。来源：decision-log 决策 16（历史坑）。
  - **场景**：Given 阶段重开产生多窗口，When 汇总，Then 阶段内先去重再求和，数值不翻倍。
- **FR-GUARD-004**：依赖会话级来源的字段，须先确证来源标识指向可达的真实会话且降级载体在磁盘可观测，方可纳入；否则标缺口。来源：decision-log 决策 16（历史坑）。
  - **场景**：Given 会话源标识指向不可达，When 采集该字段，Then 标缺口而非填零。

### 交付与持续集成（CI）

- **FR-CI-001**：结构校验与聚合脚本冒烟须纳入 workflowhub 自身的持续集成。来源：decision-log 决策 14。
  - **场景**：Given workflowhub CI，When 跑冒烟，Then 含结构校验 + 聚合冒烟。
- **FR-CI-002**：CI 中的冒烟须用临时目录注入全局路径，既覆盖全局双写代码路径又不依赖用户级目录；冒烟验收核心是结构校验通过 + 聚合逻辑正确。来源：decision-log 决策 14。
  - **场景**：Given CI 环境无用户级目录，When 冒烟注入临时目录作全局路径，Then 全局双写路径被覆盖且冒烟通过。

## 5. 模块划分

> 只说大概分几块、各管什么、怎么配合，不写实现类名。

### 采集核心

- **负责什么**：接收"哪个技能 / 什么时机 / 什么数据"，把执行效果记成"一条记录三段更新"，双写任务级 + 全局。
- **对外接口**：宿主适配层在三个时机调它落骨架 / 补自身 / 补全局影响。
- **依赖谁**：不依赖任何特定执行宿主（解耦）；只依赖存储位置配置。
- **测试边界**：深模块，可独立测（喂数据进、查记录出，无需搭宿主）。

### 宿主适配层

- **负责什么**：把特定执行宿主的触发时机翻译成对采集核心的三时机调用。
- **对外接口**：被宿主钩子触发。
- **依赖谁**：依赖采集核心 + 宿主特定机制。
- **测试边界**：需搭宿主环境，薄层。

### 统一执行记录组装

- **负责什么**：把进度/指标/反馈/边界决策/回溯索引六类键串成全景，做结构校验。
- **对外接口**：读各类源的索引，产出统一记录实体。
- **依赖谁**：依赖采集核心产出 + 卡点卡片。
- **测试边界**：可独立测结构校验。

### 卡点卡片记录

- **负责什么**：卡点出现即记结构化卡片（受控枚举类型 + 必填项 + 可选项）。
- **对外接口**：执行中检测到摩擦时调它。
- **测试边界**：可独立测 schema 校验。

### 基线与版本规则

- **负责什么**：基线字段定义 + 从历史归档算基线（缺口诚实标记）；技能版本号规则文档。
- **对外接口**：基线聚合读历史记录；版本规则供人/工具判定版本级别。
- **测试边界**：基线聚合可独立测；版本规则是文档，靠复算一致性验证。

### CI 冒烟

- **负责什么**：临时目录注入全局路径，跑结构校验 + 聚合冒烟。
- **测试边界**：CI 环境。

## 6. 关键实体（涉及数据时必填）

- **执行记录**：
  - `执行标识`：唯一定位一次技能/阶段执行，必填，关联三时机与回溯索引。
  - `技能或阶段名` / `所在阶段`：归属。
  - `技能版本`：聚合时区分版本。
  - `是否执行` / `用量` / `耗时` / `返工轮次` / `人工介入`：核心效果字段（无审查时返工轮次须可表达"无审查"区别于"零返工"）。
  - `卡点引用`：指向卡点卡片。
  - `任务标识` / `项目标识`：全局记录必带，供跨任务/跨项目聚合。
- **卡点知识卡片**：
  - `卡点类型`：受控枚举（已知类型 + 其他兜底）。
  - `发生位置` / `根本原因` / `解决方式` / `是否真正解决` / `发生时间`：必填。
  - `可迁移建议` / `关联历史教训` / `绕过非根治标记` / `关联指标引用` / `受影响交付物`：可选。
- **统一执行记录**：
  - 六类键：`进度` / `事实` / `指标` / `反馈` / `边界决策` / `回溯索引`；每类键标注首个消费方，无消费方者可选。
- **基线字段**：
  - `字段映射` / `缺口状态` / `缺口原因` / `局限说明`。

## 7. 数据和生命周期（数据驱动需求时必填）

- **数据粒度**：一条执行记录 = 一次技能执行或一个阶段。
- **数据时效**：分三时机增量更新同一条记录，非实时单写。
- **补传策略**：会话压缩后凭执行标识从磁盘重定位续写；写入失败发告警，事后从记录缺口可发现。
- **预览 vs 正式**：无预览态，落盘即正式记录。
- **当前 vs 历史**：同一执行的记录三段更新为合并覆盖（凭执行标识），跨执行为累加；全局汇总累加，跨任务保留。

## 8. 兼容性预留（分阶段交付时必填）

- **命名预留**：统一执行记录的六类键命名不堵死后续 M14 消费迭代的新增维度。
- **容器预留**：每类键标注消费方 + 预留扩展位，后续里程碑加键不构成破坏性变更。
- **状态预留**：基线字段含"缺口状态"，后续补上数据源后可由缺口转为数值，无需改结构。

## 9. 不做和隐性必达

### 明确不做

> 继承 intake 批准的每一条 out-of-scope，逐条列出。

- 不做趋势分析、优先级排序器、消费迭代入口（属后续里程碑 M14）。
- 不做自动契约比对（schema-diff）版本守卫。
- 不修改 agenthub 既有实现；本期交付物全为 workflowhub 新增。
- 不把采集核心硬绑某一特定执行宿主的钩子机制。
- 不引入对执行效果的人工主观打分（效果/关键性靠可转硬的客观数间接反映）。

### 隐性必达（默认就得满足的条件）

- 指标只记不挡（贯穿所有功能域，见 FR-GUARD-001）。
- 每条执行记录可凭执行标识唯一定位（见 FR-COLLECT-004）。
- 缺口一律诚实标记、不伪造数字或以零充缺（见 FR-EXECREC-004 / FR-BASELINE-002 / FR-GUARD-004）。
- 三个历史坑须在设计时堵死（动作计数归零 / 阶段重开翻倍 / 会话源永空，见 FR-GUARD-002/003/004）。

**可判真伪的验收标准（success criteria）**：

- **AC1**：跑一个技能，任务级与全局两处各出现一条该次执行记录；删去任一处即判失败。
- **AC2**：在已知有 N 条记录跨 M 个任务、覆盖技能 S 的版本 V 的全局汇总上，按"技能名 S + 版本 V"聚合，须返回恰好那 N 条记录所属的任务集合（分母 = 该 S+V 在全局汇总中的实际记录数 N）；**负向**：按一个全局汇总中不存在的"技能名 + 版本"组合聚合，须返回空集而非误匹配其他版本的记录。任一方向不符即判失败。
- **AC3**：一份统一执行记录能通过结构校验，六类键齐全；缺任一类键即判失败。
- **AC4**：给定一次明确的技能契约变更（如新增一个必填参数），两名独立判定者依版本规则各自判定该升哪级版本，结论须一致；**负向**：给定一次不改变对外契约的变更（如仅改注释/内部实现），版本规则须判定不触发版本号变化。任一方向判定不一致或误升即判失败。
- **AC5**：核心字段就位（含执行标识），原 5 项任务级指标能从核心字段聚合算出；缺任一核心字段即判失败。
- **AC6**：一张卡点卡片能通过结构校验（类型为受控枚举之一、必填项齐全）；缺必填项即判失败。
- **AC7**：用**已归档的 AgentHub M1-M3 任务记录**（须在基线输出中记录所用的具体源任务集合）算基线，能算的给数值、算不出的给结构化缺口（含状态/原因）；**负向**：出现伪造数字、以零充缺、或未记录源任务集合（基线不可复算）即判失败。
- **AC8**：三个防错点各有反向用例——故意构造"按消息标识去重→归零""阶段重开→翻倍""会话源不可达→填零"三种坏情况，验证系统分别按动作标识去重、阶段内先去重再求和、标缺口而非填零；反向用例不触发即判防错失效。
- **AC9**：扫描实现，作为拦截推进的硬门的指标/卡点/版本门数量为零。
- **AC10**：CI 用临时目录跑结构校验 + 聚合冒烟通过；**负向**：故意喂一份缺类键的统一执行记录或一条坏 schema 的聚合输入，CI 冒烟须判失败（红），证明冒烟不是恒绿空跑。正向通过 + 负向能红，两者都满足才算 CI 守得住。
- **AC11（行为验证）**：构造一次"采集写入失败"场景，验证系统发可观测告警且该技能/阶段仍推进（运行时行为，非仅代码存在）；**负向**：在采集写入正常的对照场景下，验证不误发写入失败告警。失败场景不报警、或正常场景误报警，任一即判失败。

- **用量字段的唯一权威来源**：存在两条候选来源，须择一并说明，留 plan 定。

# 功能规格：M2 — workflowhub 微内核核心 + 反宿主依赖规则 + 全局配置 + 扩展性验收 + path guard

> 需求权威源：本 task 的 decision-log（14 条已批准决策）。本 spec 每个概念可回指对应决策编号。

## 速读卡（30秒看懂这个需求）

给 workflowhub 装一个"薄薄的调度大脑"骨架：读一份全局配置、按清单找到组件、把组件调起来、收回结果——核心不懂任何业务。配套四样守卫：一份全局配置（多类键）、一个防止核心偷偷依赖外部环境的检查、一个防改错文件的保护、一套"加新组件不用改核心"的扩展性验收。全部能被自动检查跑出"正常绿、故意改坏红"。不做真业务、不定最终契约、不做指标消费——那些留给后面的里程碑。

## 1. 问题陈述

上一里程碑只交付了 workflowhub 的规范底座（设计宪法、文档、自动检查、空目录骨架），没有任何能把工作流跑起来的"调度核心"。没有这个核心，后续任何工作流、任何阶段都无法被装载和执行。本里程碑要造出这个核心的**骨架级版本**，并立好它与外部组件之间的边界守卫。（承接 决策 1）

## 2. 背景、目标和边界

### 现有系统
workflowhub 仓已有设计宪法、检查清单、项目文档、CI 骨架、以及 config/workflows/skills 三个空目录。已有一套机器结构校验脚本（收集错误、非空即非零退出）和 CI（克隆后安装即可跑最小检查）。

### 目标
- 造出薄核心：能读全局配置、按组件清单找到并调起组件、收回组件的结构化结果，核心不含任何业务分支。（决策 1）
- 立一份全局配置，能被核心解析出约定的多类键。（决策 8、13）
- 立三道守卫：反宿主依赖检查、禁改文件保护、路径解析单一入口。（决策 5、6、7）
- 立可证伪验收：可换性、扩展性各自独立，每件配"故意改坏必失败"的检查，全部纳入 CI。（决策 4、9、10、11）

### 明确不做（边界）
- 不实现任何真实工作阶段的业务逻辑（留 M6 及以后）。（决策 1、12）
- 不定组件输入输出的最终字段契约（留 M3）。（决策 1、14）
- 不消费"指标路径""命令行工具映射"两类配置的值（留 M4/M17）。（决策 8）
- 不做质量裁决门、不做防伪处置流程（留 M5）。
- 详见第 9 章。

## 3. 用户场景和用例

> 用户 = workflowhub 的开发者（自举者）。以下场景描述"开发者/系统在真实场景里能观察到的行为"，是后续验收的来源。

1. **正常·调起组件**：开发者在全局配置里登记一个组件，运行核心并指定一个工作流标识，核心能找到该组件、把它调起来、收回它写出的结构化结果，并能从结果里看出"是哪个组件被调起的"。（决策 1、14）
2. **正常·换组件不改核心**：开发者把配置里某个组件指向一个替身（假组件），核心代码一行不改，重新运行，核心调起的是替身，结果里能看出"调的是替身"。（决策 11 可换性）
3. **正常·加组件不改核心**：开发者在配置和目录里新增一个全新的占位组件，只通过工作流标识触发，核心代码一行不改就能把它发现并调起。（决策 4、11 扩展性）
4. **正常·解析全局配置**：开发者运行核心，核心能从全局配置读出"组件清单"这一必备项；其余几类键若存在则被检查格式。（决策 8、13）
5. **边界·缺必备配置项**：开发者删掉配置里的"组件清单"必备项，核心解析时报错、非零退出，不静默放过。（决策 8）
6. **边界·占位键缺失**：开发者删掉配置里"指标路径""命令行工具映射"这类占位键，核心不报错（这些键 M2 不消费其值）。（决策 8）
7. **失败·组件失败**：被调起的组件以非零状态结束，核心如实把它当失败，不把失败静默成成功。（决策 1）
8. **失败·组件输出非法**：被调起的组件写出的结果不是合法结构化数据，核心如实报错。（决策 1、14）
9. **守卫·反宿主命中**：开发者在核心代码里写下一处"写死的本机绝对路径 / 向上层层翻目录找根 / 依赖特定外部宿主目录布局 / 绑死特定 AI 宿主钩子语义"，反宿主检查扫描后命中、非零退出。（决策 5）
10. **守卫·反宿主干净**：核心代码不含上述任何一类，反宿主检查零退出通过。（决策 5）
11. **守卫·禁改文件命中**：开发者对一个被列入禁改清单的文件发起改动，禁改保护扫描这次改动、扫出并报错。（决策 6）
12. **守卫·禁改文件放过**：开发者改动一个不在禁改清单里的文件，禁改保护放过、零退出。（决策 6）
13. **守卫·路径解析单一入口**：核心需要解析任何路径时，都经同一个解析入口；该入口拒绝"从当前目录猜""向上找宿主""依赖外部宿主目录布局"等方式。（决策 7、13）
14. **可证伪·自检红**：每道检查都带一个"已知坏样本"，开发者运行该检查的自检，坏样本能让被测检查报红（非零），而自检本身不把整体检查流程弄成永久红。（决策 9、10）
15. **CI·完整门**：开发者提交改动，CI 跑全部检查，结果是"正常样本全绿 + 各检查的坏样本自检证明能报红"。（决策 10）

## 4. 功能需求

### 域 CORE — 薄核心调度

- **FR-CORE-001**：核心必须能读取全局配置、解析出组件清单，并按给定的工作流标识从清单定位到组件。（决策 1、8）
  - 验收（Given/When/Then）：给定一份含组件清单的配置和一个有效工作流标识，当运行核心时，则核心能定位到对应组件并尝试调起它。
- **FR-CORE-002**：核心调起组件采用黑盒契约——组件是"可被核心调用、结束时返回成功/失败状态、写出一份合法结构化结果、结果中含组件身份标识"的黑盒；核心不规定组件内部如何实现，也不规定数据在核心与组件间用何种方式传递（传输形态留 M3）。（决策 1、14）
  - 验收：给定一个符合黑盒契约的组件，当核心调起它，则核心收回其结构化结果，能从中读出组件身份标识，并据返回状态判定成功或失败。
- **FR-CORE-003**：核心必须不含任何业务分支——不得出现按具体阶段名/业务类型分流的判断；新增、替换组件都只经配置与组件清单，核心代码不变。（决策 1、12）
  - 验收：给定核心代码，当扫描其内容，则不存在按具体阶段名/业务类型硬编码的条件分流。
- **FR-CORE-004**：组件以非零状态结束、或写出的结果非合法结构化数据时，核心必须如实判为失败/报错，不得静默成功。（决策 1）
  - 验收：给定一个返回非零状态或写出非法结果的组件，当核心调起它，则核心判为失败/报错并以非零状态反映。

### 域 CFG — 全局配置解析

- **FR-CFG-001**：全局配置必须含"组件清单"这一必备键；缺失时核心解析报错、非零退出。（决策 8）
  - 验收：给定一份缺少"组件清单"键的配置，当核心解析，则报错并非零退出。
- **FR-CFG-002**：全局配置的其余约定键（外部依赖声明、指标路径、命令行工具映射、禁改清单）采用"存在即校验形状、缺失不报错、不校验其值语义"的处理；其中指标路径、命令行工具映射两类键 M2 不消费其值。（决策 8、13）
  - 验收：给定一份这些键存在但形状错误的配置，当核心解析，则报错；给定这些键缺失的配置，当核心解析，则不报错。
- **FR-CFG-003**：全局配置只承载静态配置；任何"运行时当前状态"不得进入全局配置（避免全局单指针被并发写而串台）。（决策 8）
  - 验收：给定全局配置的约定键集合，当审视其语义，则不含任何运行态/当前活跃态字段。
- **FR-CFG-004**：任务执行记录目录（框架配置）必须作为一项框架配置有落点，经路径解析单一入口解析；M2 只做声明与解析，不消费其运行态语义（执行记录如何写、缺失运行时行为留 M3+）。（决策 13）
  - 验收：给定一份声明了任务执行记录目录的配置，当核心解析，则该项经统一入口被解析为路径，核心不读写其运行态内容。

### 域 GUARD — 反宿主依赖检查

- **FR-GUARD-001**：必须提供一个反宿主依赖检查，扫描**核心本体**（由核心边界规则界定的范围，不含守卫脚本自身——守卫脚本是工具、天生要触及路径与环境，纳入扫描会误报），命中以下任一类即报错、非零退出：①写死的本机绝对路径；②向上层层翻目录寻找仓库根；③依赖特定外部宿主的目录布局/派生路径；④绑死特定 AI 宿主的钩子语义。（决策 5；Clarifications 2026-06-22）
  - 验收：给定核心代码植入上述任一类，当运行检查，则非零退出并指出命中项；给定干净核心代码，当运行检查，则零退出。
- **FR-GUARD-002**：反宿主检查必须自带"已知坏样本"自检，能证明检查在"实际命中"时确实报红，而非空跑通过。（决策 9）
  - 验收：给定检查的坏样本自检，当运行，则被测检查对坏样本报红（非零）。

### 域 PATHG — path guard 与路径解析

- **FR-PATHG-001**：必须提供禁改文件保护——基于一份声明式禁改清单，扫描给定的改动，对命中禁改清单的改动报错；对未命中的放过。保护必须可靠覆盖直接的文件编辑/写入类改动。（决策 6）
  - 验收：给定一次对禁改清单内文件的改动，当运行保护，则扫出并报错；给定一次对清单外文件的改动，当运行保护，则放过。
- **FR-PATHG-002**：禁改保护必须幂等，且在任何情况下都不主动写回/复活文件（避免保护机制自身造成流程死结）。（决策 6）
  - 验收：给定连续多次运行保护，当对比运行前后，则保护自身不产生任何文件写入。
- **FR-PATHG-003**：禁改保护对无法可靠覆盖的旁路改动方式（如经由内联文档块、动态变量、外部脚本写文件）必须如实声明为"已知不覆盖"，不假装覆盖。（决策 6）
  - 验收：给定保护的能力声明，当审视，则明确列出已知不覆盖的旁路方式。
- **FR-PATHG-004**：核心解析任何路径都必须经同一个解析入口；该入口拒绝"从当前工作目录推断""向上寻找宿主根""依赖外部宿主派生布局"等方式。（决策 7、13）
  - 验收：给定核心中任意需要路径的位置，当审视其取路径方式，则均经统一入口，且不存在被禁止的推断方式。

### 域 EXT — 扩展性与可换性验收

- **FR-EXT-001**：必须提供可换性验收——把配置里某组件指向一个替身，核心代码不变即能调起替身，且能从结果里证明"调起的是替身"。核心代码若被改动则验收失败。（决策 11）
  - 验收：给定把组件指向替身的配置，当运行可换性验收，则核心不改动即调起替身、结果含替身身份标识；若核心被改动则失败。
- **FR-EXT-002**：必须提供扩展性验收——新增一个占位组件，仅通过工作流标识触发（不手工传入组件位置），核心代码不变即能发现并调起它。核心代码若被改动则验收失败。（决策 4、11）
  - 验收：给定新增的占位组件，当仅以工作流标识运行扩展性验收，则核心不改动即发现并调起它；若核心被改动则失败。
- **FR-EXT-003**：可换性与扩展性必须是两件独立的验收（替换 vs 新增），不得合并为一件。（决策 11）
  - 验收：给定两套验收，当审视，则分别覆盖"替换已有组件"与"新增组件"两种操作。

### 域 CI — 验收闭环与可证伪

- **FR-CI-001**：核心目录边界由一条单一、机器可读、可复用的边界规则界定（决策 4 批准的 design 硬门，非普通实现细节——它是三项验收可证伪性的前置真相源）。边界规则形状（原型，schema 形状性表达，非实现代码）：
  - 核心本体 = 单层目录 `core/` 下的全部实现文件，匹配规则 `core/**/*.mjs`；
  - 非核心（不受核心 diff/反宿主约束）= 配置 `config/`、组件 `workflows/`、组件 `skills/`、守卫与验收脚本 `scripts/`、测试夹具 `fixtures/`，各为顶层单层目录；
  - 该规则是 diff（可换性/扩展性判"核心是否被改动"）、反宿主扫描（判扫描范围）、漏扫自检三者共用的**唯一锚点**，禁止各处各写一份（防漂移）。
  可换性、扩展性、反宿主三项验收都以 `core/**/*.mjs` 界定的范围为锚点；新增一个属于核心范围的文件若被该规则漏掉，验收必须失败。（决策 4）
  - 验收：给定一个新增的、本应属于核心范围的文件（落在 `core/` 下），当边界规则未覆盖它，则漏扫自检失败（漏扫可被检出）；给定核心范围外的文件改动，则不触发核心 diff 失败。
- **FR-CI-002**：每项验收/检查都必须带一个"故意改坏必失败"的变异自检；自检在被测对象被改坏时报红，且自检本身不把整体检查流程变成永久红（坏样本由自检在子过程中验证，主流程仍可正常通过）。（决策 9、10）
  - 验收：给定每项检查的变异自检，当运行，则坏样本让被测检查报红，而整体检查流程对正常样本仍绿。
- **FR-CI-003**：上述全部检查（反宿主、禁改保护、可换性、扩展性、配置解析、各自变异自检）必须纳入项目的统一检查入口并由 CI 自动运行；完成标准是"正常样本全绿 + 各检查坏样本自检证明能报红"，而非"脚本被执行过"。（决策 10）
  - 验收：给定 CI 运行，当检查全套，则正常样本全绿、坏样本自检证明报红能力。

## 5. 模块划分

> 只标模块边界、接口形状（窄）、测试边界；不写文件路径、不写实现代码。

| 模块 | 职责（窄接口） | 测试边界/层级 |
|------|---------------|---------------|
| 调度核心 | 读配置→按清单定位组件→调起→收结构化结果→判成败。只认契约形状不认业务 | 行为测试（P0）：调起/换/失败/非法输出 |
| 配置解析 | 把全局配置解析成约定键集合；必备键缺失报错，其余 shape-only | 单元+行为（P1）：缺必备键红、占位键缺失绿 |
| 路径解析入口 | 所有路径取值的单一入口；拒绝宿主推断 | 单元（P1）：禁止推断方式被拒 |
| 反宿主检查 | 扫核心代码四类宿主耦合，命中非零 | 自检变异（P0）：坏样本必红 |
| 禁改保护 | 按禁改清单扫改动，命中报错，幂等不写回 | 自检变异（P1）：越界改动必红、幂等 |
| 可换性/扩展性验收 | 替换/新增两件独立验收，核心 diff 锚定边界规则 `core/**/*.mjs`（与反宿主、漏扫自检共用同一锚点，不各写一份） | 集成（P0）：核心改动则失败 |

测试方法标准对照（风险分级/可追溯/三层节奏/发布门）遵循平台测试蓝本；本 feature 整体属"单一工具内的核心+脚本"类，收尾路由在 apply 完成时判类。每条 FR 的行为验收即 test-acceptance 的来源。

## 6. 关键实体（涉及数据时必填）

- **全局配置**：静态配置容器。约定键——组件清单（必备）、外部依赖声明、指标路径、命令行工具映射、禁改清单；外加框架配置项任务执行记录目录。行为相关属性：哪些键必备（缺失报错）、哪些 shape-only（存在校形状、缺失放过、不消费值）。（决策 8、13）
- **组件清单条目（registry entry）**：把"工作流标识 → 组件"映射起来的最小登记项。行为相关属性：能被核心据工作流标识定位到、指向一个可被调起的组件。M2 不固定其内部字段表（留 M3）。（决策 1、14）
- **组件结果（黑盒输出）**：组件被调起后写出的结构化结果。行为相关属性：合法结构化、含组件身份标识、伴随成功/失败状态。M2 只固定"含身份标识 + 合法结构化 + 有状态"，不固定其余字段（留 M3）。（决策 1、14）

## 7. 数据和生命周期（数据驱动需求时必填）

本期不涉及持久化数据存储或数据时效/补传。全局配置是静态读入；组件结果是一次性结构化产物，核心读取后据以判断推进，不在本期定义其留存生命周期（运行态记录语义留 M3+）。（决策 13）

## 8. 兼容性预留（分阶段交付时必填）

本期是骨架级，为后续里程碑显式预留接口而不实现其语义：
- 组件结果只固定"身份标识 + 合法结构化 + 状态"，其余字段表与传输形态留 M3 的窄契约定版。（决策 1、14）
- 配置中"指标路径""命令行工具映射"两键只声明与校形状，其值消费分别留 M4、M17。（决策 8）
- 任务执行记录目录只声明与解析，运行态消费语义留 M3+。（决策 13）
- 工作阶段/组件的真实命名留 M6/M8；核心对阶段名透明、零硬编码，命名是配置里的数据。（决策 12）

## 9. 不做和隐性必达

### 不做（out-of-scope）
- 真实工作阶段的业务逻辑（→ M6 及以后）。（决策 1、12）
- 组件输入输出最终字段契约定版（→ M3）。（决策 1、14）
- 指标路径/命令行工具映射的值消费（→ M4/M17）。（决策 8）
- 质量裁决门、防伪处置流程（→ M5）。
- 禁改保护抓全部旁路（内联文档块/动态变量/外部脚本写文件列为已知不覆盖）。（决策 6）

### 隐性必达
- 平台约束交叉比对结论：14 条决策对 workflowhub 宪法/术语表/项目硬规则无冲突。M2 的"必红/非零退出"属宪法允许的机器入口校验/可证伪结构检查，非被禁的主观质量门；反宿主检查与路径单一入口反而是"可搬运""薄核心边界"原则的强制落地。
- 术语补充：术语表当前未定义"组件 / 组件身份标识 / 组件清单"，本 spec 已在第 6 章给出行为定义；后续若编辑术语表，新增词需避开既有机器校验的排除词表。
- 所有检查在"实际为假"时必须真实报红（可证伪），不得假绿；新增防假绿检查须先经变异自证（故意改坏看是否真红）。（决策 9）

## 11. 影响范围（业务性质）

本里程碑在一个全新的独立项目仓内新增核心与守卫，**不改动任何已有功能**（上一里程碑只交付了规范底座与空目录骨架，本期是首次填入核心）。
- 受影响的已有产物：上一里程碑的统一检查入口与 CI——本期向其新增多项检查（反宿主/禁改/可换性/扩展性/配置解析及各自变异自检），属新增挂载，不改其既有的结构校验行为。
- 需回归路径：上一里程碑既有的结构校验在新增检查后仍须正常通过（正常样本不被新检查误伤）。
- 对其他里程碑：为 M3（窄契约）、M4（指标消费）、M5（质量门）、M6/M8（真阶段与命名）、M17（多 CLI）显式预留接口，本期均不实现其语义。
- 无影响：不触及任何外部用户可见功能（workflowhub 是开发者自举工具，本期产物面向自举开发者）。

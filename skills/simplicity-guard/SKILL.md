# simplicity-guard

## 定位

横切组件技能，不产 stage-result，不单独计指标。由顶层 skill（build-spec、build-plan）在需要做复用判断时调起，结论写入调用方指定字段。

## 四阶梯决策树

依次向下，遇到第一个"能用"就停，不继续往下问。

```
P0：这东西需要存在吗？
    → 不需要（YAGNI）：跳过，不写任何代码
    → 需要：继续 P1

P1：已有覆盖？
    仓库里已有能直接用的 / 标准库能做 / 平台原生功能能做 / 已安装依赖能做
    → 任一为真：直接用，禁止重写
    → 全否：继续 P2

P2：复用+改造？
    已有的东西改一改能不能用（改造也算复用，不算重写）
    → 能：改造复用，只改必须改的部分
    → 不能：继续 P3

P3：最小新增
    前三层都不行，才写代码。
    写法纪律（三条必须全部遵守）：
    ① 先想后写：动手前说清楚为什么 P0-P2 都不行，不确定就停下问，不做静默假设
    ② 最小代码：只写解决问题所需的最少代码，不写投机功能
    ③ 手术式修改：只动必须动的地方，只清理自己改动产生的孤儿代码，每行改动可追溯到需求
```

## 产物字段

调用方在其产物中写入 `minimal-path` 字段，格式：

```
minimal-path: P<N> — <一句话结论>
示例：
  P1 — 直接用标准库 fs.readFile，不新增依赖
  P2 — 改造现有 spec-clarify 加参数，不重写
  P3 — 现有机制不覆盖，新增最小实现（先想后写/最小代码/手术式修改）
```

## 例外（不适用四阶梯）

以下场景绝不简化，即使四阶梯指向"跳过"也不能省：

- 信任边界的输入校验
- 防丢数据的错误处理
- 安全措施
- 无障碍基础

## 配套禁止项

- 禁止投机性抽象（为"将来可能用到"而写的代码）
- 删除优于新增（能删就不要保留）
- bug 走根因，不打补丁
- 说明文字不能比代码还长

# talk-with-zhipeng

> 一个通用的「对话收敛」技能。目标：方向 / 歧义 / 风险 / 范围 四件事在同一场大白话对话里收敛，产出结构化决策记录。

本技能分两层：

- **平台无关核心层**：纯对话收敛逻辑，包括调研摆出、影响排序提问、每答重排、收敛阈值、四维判定、输入护栏、丢弃登记、歧义校正。
- **薄适配层**：只负责对接当前 workflowhub 工作流的产物读写。默认从当前任务的调研与规格材料读取背景，把输出写回当前任务约定的相对路径产物。

判别器：一段逻辑是否依赖「具体工作流产物的读写」？依赖则入薄适配层；不依赖则入核心层。核心层不得触碰任何工作流私有路径。

## 输入

- 已有调研、规格、计划或代码阅读摘要。
- 用户原始需求、补充回答和可引用的上下文。
- 当前工作流约定的相对路径产物，例如 `artifacts/`、`specs/<task>/`、`tasks/<task>/decision-log.md`。

若输入缺失，不直接提问，先进入输入充分性护栏并说明缺什么。

## 执行协议

### 1. 启动先摆调研

启动后先不要提问。先把已有调研结果整理成一份「初始咨询材料」摆给人看：调研摘要 + 待澄清问题集，每个问题带「当前影响权重」。让人先看清你已经知道什么，再开始问。

若调研结果缺失，先自检并反馈缺什么，不带着空背景进入追问。

### 2. 影响排序提问

把所有待问问题按「对方向的影响」从高到低排序。影响标尺 5 档：

| 档位 | 权重 | 含义 |
|------|:--:|------|
| high | 5 | 答案直接决定方向 |
| medium | 4 | 答案明显改变方向取舍 |
| low | 3 | 答案只影响细节，不动方向 |
| none | 2 | 答案对方向无关 |
| 已答 | 1 | 已经确定，不再问 |

一次只问一个问题，即当前影响最高的问题，并说清「这个问题不同答案会带来什么后果」。影响主流程的问题在聊天里问；不影响主流程的问题做最小合理假设并说明，不打断人。

### 3. 每答重排

每收到一个回答，就用最新信息重新计算剩余问题的影响排序，再决定下一个问。不要按固定顺序往下走，新信息可能让原本低影响的问题升成高影响。

### 4. 收敛阈值

每答重排后判断是否收敛。收敛判据：剩余问题里「无 high 且无 medium」即收敛。方向已无大悬念后停止提问，产出决策记录。

该判据复用影响排序机制，不引入第二套评分系统。

### 5. 范围四维判定

范围判断融入同一场对话。四个维度每维结论必须标「证据」还是「主观」：

- **A 真实痛点**：用户原文能直接引用卡在哪 = 证据；从行为推导、非用户原话 = 主观。
- **B 复杂度与 ROI**：有参照 case、改动量可估、ROI 可量化 = 证据；首次做、工程量靠猜、ROI 无数据 = 主观。
- **C 风险与影响范围**：能指认改动边界或列出受影响模块名 = 证据；无法列出具体影响路径 = 主观。
- **D 时机**：项目计划或阶段目标能引用 = 证据；有顾虑但无明确冲突 = 主观。

四选一裁决：可以做 / 可做但缓一缓 / 有风险需限制范围 / 不建议做。附上推翻条件清单，说明满足哪些条件时裁决会改变。

轻量档短路：当范围已明确且显然无需展开四维时，可跳过完整四维分析，直接输出 `verdict=轻量档可忽略`。

### 6. 输入充分性护栏

四维分析之前先自检输入是否够：

1. 需求描述是否足够具体，能判断痛点真实性？
2. 项目背景是否足以评估影响范围？
3. 时机相关上下文是否存在？

缺哪类信息且无法从上下文推断，直接反馈缺什么，不往下跑：

- 缺痛点描述：`缺少用户原始痛点描述，请补充`
- 缺影响范围线索：`缺少项目代码或模块背景，影响范围无法评估`
- 时机完全空白：可降级为「主观」继续，但必须在输出中标注

对需要推断、无法从原文直接读出的前提，在输出前显式回显让人纠正。

### 7. 丢弃台账登记

判定丢弃任何一条原始需求条目，必须登记「理由 + 去向」，禁止静默丢。理由缺失时裁决不成立。

### 8. 大白话聊天

所有提问与回答用大白话、走聊天交互。一次最多问一个关键问题，其余走假设并说明。只有用户明确要求选择器时才使用结构化选项。

### 9. 歧义校正与方向收敛并行

回答含歧义或自相矛盾时，当场指出歧义并追问澄清，不带着歧义往下走。歧义校正与方向收敛并行进行。

### 10. 结构化字段显式保留

四维结论、四选一裁决、推翻条件、每条结论的「证据/主观」标注，都以结构化字段显式保留，落到决策记录里。

## 输出

```text
## 决策记录
- 最终 verdict：[可以做 | 可做但缓一缓 | 有风险需限制范围 | 不建议做 | 轻量档可忽略]
- 四维详情：
  - 真实痛点：[正面/中性/负面]（来源：证据/主观）
  - 复杂度/ROI：[正面/中性/负面]（来源：证据/主观）
  - 风险与影响范围：[正面/中性/负面]（来源：证据/主观）
  - 时机：[正面/中性/负面]（来源：证据/主观）
- 推翻条件：[什么情况下这个裁决会改变]
- 丢弃台账：[逐条 原始需求 -> 丢弃理由 -> 去向]
- 推断回显：[对哪些前提做了推断，待用户纠正]
```

## workflowhub 薄适配层

核心层产出的是「调研入 / 决策记录出」的纯逻辑。薄适配层只做两件事：

1. **读调研**：从当前 workflowhub 任务约定的相对路径读取调研、规格、计划或测试证据，整理成初始咨询材料。
2. **落记录**：把结构化决策记录写入当前任务约定的相对路径产物，例如 `tasks/<task>/decision-log.md` 或当前 issue 指定的 artifact。

适配层只负责「从哪读、往哪写」，不含对话收敛判断逻辑。换一个工作流只需换这一层，核心层不动。

## 约束

- 每条四维结论必标「证据」或「主观」，不得遗漏；「证据」必须有具体原文引用或可验证数据来源。
- 一次最多问一个关键问题，其余走最小合理假设并说明。
- 核心层不触碰任何具体工作流的私有路径。
- 不修改与对话无关的文件；丢弃任一需求条目必须登记理由。
